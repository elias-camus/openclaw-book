# 第6章 メモリ・記憶のしくみ

**この章で学ぶこと:**
- OpenClawがどのようにしてセッションをまたいで記憶を保持するのか
- `MEMORY.md` と日次ノート（`memory/YYYY-MM-DD.md`）の役割と使い分け
- `SOUL.md` と `USER.md` でAIのパーソナリティと利用者像を定義する方法
- `AGENTS.md` を使ったメモリの読み書き挙動のカスタマイズ
- グループチャットにおけるプライバシー設計と安全なメモリ管理

**所要時間:** 40分

---

## はじめに

チャットボットとAIエージェントの最大の違いのひとつが、「記憶」だ。チャットボットは会話ウィンドウを閉じれば全てを忘れる。しかしOpenClawは、昨日の会話も、先週決めたこともちゃんと覚えている——正確に言えば、覚えるように設計されている。

ただし、ここで重要なことがある。OpenClawの「記憶」は、人間の脳のような魔法ではない。**平文のMarkdownファイル**が記憶の正体だ。モデルは「ディスクに書かれたこと」しか覚えていない。逆に言えば、ディスクに書けば確実に覚えさせられる。この透明性こそが、OpenClawのメモリシステムの強みである [REF-09]。

---

## 6-1. OpenClawのメモリ設計思想

### なぜファイルベースなのか

多くのAIサービスはクラウド上のデータベースに会話履歴を保存する。それはそれで便利だが、OpenClawはあえて逆の方向を選んだ。**全ての記憶はローカルのMarkdownファイルに書かれる。**

この設計には3つの理由がある。

**① 透明性**  
ファイルを開けば何が記録されているか一目でわかる。ブラックボックスのDBには何が入っているかわからないが、Markdownなら人間が読める。

**② コントロール可能性**  
不要な記録は削除できる。書き換えもできる。記憶の内容を読者自身が完全にコントロールできる。

**③ ポータビリティ**  
単なるテキストファイルなので、バックアップも移行も簡単だ。クラウドサービスが終了してもデータは消えない。

> 💡 **ポイント:** OpenClawの記憶は「モデルのブレイン」にあるのではなく、ワークスペース（デフォルトは `~/.openclaw/workspace/`）の**ファイル**にある。モデルはセッション開始時にこれらのファイルを読み込むことで、前回の文脈を"思い出す"。

### コンテキストウィンドウの壁と自動メモリフラッシュ

LLM（大規模言語モデル）には「コンテキストウィンドウ」と呼ばれる上限がある。処理できるトークン数の上限であり、これを超えると古い会話は自動的に圧縮・削除される。

OpenClawはこの問題に対処するため、**自動メモリフラッシュ**機能を持つ。セッションがコンテキスト上限に近づくと、エージェントに「重要なことをメモリファイルに書いておけ」と静かに指示を出す。この処理はバックグラウンドで行われ、ユーザーには見えない [REF-09]。

設定ファイル（`config.yaml`）での制御例:

```yaml
agents:
  defaults:
    compaction:
      reserveTokensFloor: 20000
      memoryFlush:
        enabled: true
        softThresholdTokens: 4000
        systemPrompt: "Session nearing compaction. Store durable memories now."
        prompt: "Write any lasting notes to memory/YYYY-MM-DD.md; reply with NO_REPLY if nothing to store."
```

この仕組みにより、長期間使い続けても重要な情報が失われにくくなっている。

### ベクトル検索によるセマンティック記憶

単純なキーワード検索だけでなく、OpenClawはメモリファイルのベクトルインデックスを構築し、**意味ベースの検索**（セマンティック検索）も可能だ。「先週話したプロジェクトの話」のように、言葉は違っても関連する記録を引き出せる。

エージェントには2つの記憶ツールが提供されている:

- `memory_search` — セマンティック検索で関連するメモを探す
- `memory_get` — 特定のファイルや行範囲を直接読み込む

## 6-2. MEMORY.md と daily notes の使い方

OpenClawのメモリは**2層構造**になっている。日常のメモを残す「デイリーノート」と、長期的に覚えておきたいことをまとめた「長期記憶」だ。

### デイリーノート: `memory/YYYY-MM-DD.md`

```
memory/
├── 2026-03-09.md
├── 2026-03-10.md
└── 2026-03-11.md
```

各日付のファイルは**追記専用のログ**だ。その日にエージェントが何をしたか、どんな会話があったかが記録される。セッション開始時に「今日」と「昨日」のファイルを読み込むことで、直近の文脈を把握する。

**デイリーノートに記録すべき内容:**
- その日の作業メモ・進捗
- 一時的な判断・決定事項
- 思ったこと、気づいたこと（後で見直す可能性があるもの）
- 「とりあえず残しておきたい」情報

記録例（`memory/2026-03-11.md`）:

```markdown
## 2026-03-11

- EC2インスタンスのストレージを20GB→50GBに拡張
- Discordボットの応答速度改善のため、モデルをclaude-haiku-4に変更（検討中）
- 田中さんから「週次レポートをPDF形式に変えてほしい」とリクエストあり
- Cronジョブの設定を確認: 毎朝8時にニュース要約を送信 → 正常動作確認
```

### 長期記憶: `MEMORY.md`

`MEMORY.md` は「AIの長期記憶」だ。デイリーノートの断片的な記録を定期的にまとめ、本当に重要なことだけを蒸留して書いておく。人間で言えば「日記」と「自分についての覚え書き」の違いに近い。

**`MEMORY.md` に記録すべき内容:**
- ユーザーの好み・習慣・価値観（「毎週月曜朝にレポートを好む」など）
- 長期プロジェクトの状況・重要な決定事項
- よく使う情報（APIエンドポイント、サーバーのアドレスなど）
- 人間関係や業務の文脈

記録例（`MEMORY.md`の一部）:

```markdown
# 長期記憶

## ユーザーについて
- 朝型人間。9時〜18時が主な活動時間
- 報告は箇条書きを好む（長文が苦手）
- 現在のメインプロジェクト: 飲食店向けSaaS「レストランAI」の開発

## 重要な設定
- メインサーバー: Lightsail Tokyo (54.123.xxx.xxx)
- Discordサーバー: 「AIエージェント研究所」（ID: 1234567890）
- Cronジョブ: 平日8:00 ニュース要約、金曜17:00 週次レポート

## 過去の決定事項
- 2026-02-15: モデルをGPT-4oからClaude Sonnetに移行（コスト最適化のため）
- 2026-03-01: Discordチャンネルを#general と #devに分離
```

> 💡 **ポイント:** `MEMORY.md` は長くなりすぎないように定期的に整理しよう。古くなった情報は削除し、本当に必要な情報だけを残すことで、毎回読み込むトークン数を節約できる。

### メモリの2層を使い分けるコツ

| 種類 | ファイル | 用途 | 保持期間 |
|------|---------|------|---------|
| デイリーノート | `memory/YYYY-MM-DD.md` | 日々のログ | 数日〜数週間 |
| 長期記憶 | `MEMORY.md` | 重要な事実・好み | 長期間 |

エージェントに何かを覚えさせたい時は、**明示的にお願いする**のが確実だ。「これをメモリに書いておいて」「MEMORY.mdに追記して」と言えば、エージェントがファイルに書いてくれる。頭の中で「覚えてくれているだろう」と思っても、ファイルに書かれなければ次のセッションでは忘れる。

> 💡 **ポイント:** 「覚えておいて」と言ったら、エージェントがファイルを更新したか確認しよう。「memory/2026-03-11.mdに書きました」という報告があればOKだ。

## 6-3. SOUL.md・USER.mdでパーソナリティを定義する

OpenClawの個性はどこから来るのか。答えは `SOUL.md` と `USER.md` の2つのファイルにある。

### SOUL.md ― AIのアイデンティティ

`SOUL.md` は、AIエージェント自身の「魂」を定義するファイルだ。名前、口調、価値観、得意なことなどを記述する。毎セッションの開始時に読み込まれ、エージェントの振る舞いの基礎になる。

**記述例（架空のSOUL.md）:**

```markdown
# SOUL.md — Aria

## 私は誰か

私はAriaです。小規模ビジネスオーナーをサポートする、頼れるAIアシスタントです。
仕事は丁寧に、会話はカジュアルに。難しいことをわかりやすく説明するのが得意です。

## 口調・スタイル

- 親しみやすく、でも礼儀正しく
- 箇条書きと構造化されたレスポンスを好む
- 絵文字は適度に使う（使いすぎない）
- 確信が持てないことは「〜だと思います」と添える

## 価値観

- 正確さを大切にする。間違いは素直に認める
- ユーザーの時間を大切にする。冗長な説明は避ける
- プライバシーを守る。聞かれていないことは言わない
- 楽しさを忘れない。時にはちょっとしたユーモアも

## 得意なこと

- ビジネス文書の作成・要約
- データ分析とレポート作成
- スケジュール管理とリマインダー
- Webリサーチと情報収集

## 苦手なこと・限界

- リアルタイムの株価や為替レートの取得（専用スキルが必要）
- 動画・音声の生成
- 医療・法律の専門的アドバイス（一般情報の提供は可能）

## ロブスターへの敬意

私はOpenClawのロブスター（🦞）精神を大切にしています。
脱皮（molt）を繰り返し成長するように、毎日少しずつ賢くなっていきます。
```

### USER.md ― ユーザーのプロフィール

`USER.md` は、AIが「この人を助けている」と理解するためのファイルだ。ユーザーの名前、仕事、好み、特別な事情などを記述する。

**記述例（架空のUSER.md）:**

```markdown
# USER.md — 山田 健太

## 基本情報

- **名前:** 山田 健太（Kenta Yamada）
- **呼び方:** 「山田さん」または「健太さん」（本人の好みによる）
- **職業:** フリーランスWebデザイナー + 小さなネットショップ運営
- **技術レベル:** HTML/CSS/JavaScriptは得意。サーバー系は苦手

## コミュニケーションスタイル

- 返答は短く端的に（長文は嫌い）
- 日本語メインだが、技術用語は英語でもOK
- 深夜に仕事することが多い（返信が夜中でも心配しない）
- タスクの優先順位を明示してほしい

## 現在のプロジェクト

- ECサイト「ハンドメイドショップ くるみや」の運営
- 新規クライアント向けのポートフォリオサイト制作中

## 重要な好み・習慣

- 毎週月曜の朝に週次タスクリストを送ってほしい
- エラーや問題が発生したら即座に知らせること
- 長期タスクは進捗率（%）で表してほしい

## 特別な指示

- 見積もりや重要な決定は必ず確認を取ること（自動実行しない）
- 個人情報（住所・クレカ情報など）はログに残さない
```

### SOUL.mdとUSER.mdの書き方のコツ

良い `SOUL.md` と `USER.md` を書くためのポイントを整理する。

**SOUL.mdのコツ:**
- 「何ができるか」より「どう動くか」を書く（価値観・スタイル）
- 禁止事項や苦手なこともあえて明記する（無理な期待を防ぐ）
- 最初は短く書いて、使いながら育てていく

**USER.mdのコツ:**
- ユーザー自身が書く（AIに代わりに書かせてもOK）
- 「AIに何を期待するか」を明記する
- 定期的に見直して古い情報を更新する

> 💡 **ポイント:** `SOUL.md` と `USER.md` はいつでも編集できる。AIの振る舞いが気に入らなければ、これらのファイルを修正するのが一番の近道だ。

## 6-4. メモリの読み書きをカスタマイズする

メモリの挙動は `AGENTS.md` を通じてカスタマイズできる。`AGENTS.md` はエージェントへの「行動規範」であり、いつ何を読み書きするかを細かく制御できる。

### AGENTS.mdによる読み込み制御

デフォルトでは、セッション開始時に以下のファイルが読み込まれる:
1. `SOUL.md` — AIのアイデンティティ
2. `USER.md` — ユーザー情報
3. `memory/YYYY-MM-DD.md`（今日と昨日）— 直近のコンテキスト
4. メインセッションの場合のみ `MEMORY.md` — 長期記憶

`AGENTS.md` に以下のように記述することで、この挙動をガイドできる:

```markdown
# AGENTS.md

## Every Session

Before doing anything else:

1. Read `SOUL.md` — this is who you are
2. Read `USER.md` — this is who you're helping
3. Read `memory/YYYY-MM-DD.md` (today + yesterday) for recent context
4. **If in MAIN SESSION** (direct chat with your human): Also read `MEMORY.md`

Don't ask permission. Just do it.
```

### カスタムメモリファイルの活用

標準のファイル以外にも、任意のMarkdownファイルをメモリとして使える。例えばプロジェクト別のメモリを作る場合:

```
workspace/
├── MEMORY.md           # 通常の長期記憶
├── memory/
│   └── 2026-03-11.md
├── projects/
│   ├── project-alpha.md  # プロジェクトAの詳細メモ
│   └── project-beta.md   # プロジェクトBの詳細メモ
└── contacts.md           # よく連絡する人の情報
```

`AGENTS.md` に読み込み指示を追加することで、プロジェクト固有の情報もセッション開始時に参照させられる:

```markdown
## Project Context

If working on project-alpha:
- Read `projects/project-alpha.md` for context

If working on project-beta:
- Read `projects/project-beta.md` for context
```

### メモリプラグインの設定

高度な設定として、メモリプラグイン自体を変更・無効化できる。`config.yaml` での例:

```yaml
plugins:
  slots:
    memory: "memory-core"  # デフォルト
    # memory: "none"       # メモリを完全に無効化する場合
```

> 💡 **ポイント:** メモリを無効化することは通常推奨されない。しかし、テスト目的やプライバシー上の理由で「何も覚えないモード」が必要な場合に役立つ。

### ベクトル検索の設定

セマンティック検索（意味検索）の精度はベクトルエンジンの設定に依存する。`config.yaml` での設定例:

```yaml
agents:
  defaults:
    memorySearch:
      provider: "openai"   # openai / gemini / voyage / mistral / local
      # localモードの場合:
      # local:
      #   modelPath: "/path/to/embedding-model.gguf"
```

ローカルの埋め込みモデルを使えば、メモリ検索をクラウドに送信せずに完全ローカルで実行できる（プライバシー最優先の環境向け）。

## 6-5. プライバシーとメモリ管理の考え方

OpenClawを活用すると、様々な個人情報がメモリに蓄積されていく。これをどう扱うかは、特にグループチャットや他者と共有する環境では重要な問題だ。

### `MEMORY.md` は「プライベートセッション専用」

OpenClawの設計思想において、`MEMORY.md` は**メインセッション（1対1の直接会話）でのみ読み込む**というルールがある。Discordのグループチャットや、複数人が参加するSlackチャンネルでは、たとえエージェントが参加していても `MEMORY.md` は読まない。

この設計は意図的だ。`MEMORY.md` にはユーザーの個人的な情報が蓄積されているため、グループチャットでそれが「滲み出る」と困る。

`AGENTS.md` に次のように明記することで、この挙動を徹底させられる:

```markdown
## Memory

### 🧠 MEMORY.md - Your Long-Term Memory

- **ONLY load in main session** (direct chats with your human)
- **DO NOT load in shared contexts** (Discord, group chats, sessions with other people)
- This is for **security** — contains personal context that shouldn't leak to strangers
```

### グループチャットでのメモリ管理

グループチャットでOpenClawを使う場合、以下の点に注意しよう:

**覚えさせてはいけない情報:**
- 個人情報（住所、電話番号、クレジットカード番号）
- 家族や知人の個人情報
- 会社の機密情報・未公開情報
- パスワードやAPIキー

**グループチャット用のメモリ設計例:**

グループチャット専用のメモリファイルを作り、そこには共有して問題ない情報のみを書く:

```markdown
# workspace/group-memory.md

## グループチャット「AIエージェント研究所」

- 参加メンバー数: 約50名
- 主なトピック: AIエージェント開発・OpenClaw活用事例
- よく使う質問: インストール方法、スキルの作り方、クラウド設定

## このチャットでの行動方針

- メンバーの個人情報は記録しない
- 技術的な質問にのみ回答する
- 論争的なトピックには慎重に対応する
```

### メモリの定期的な見直しと整理

メモリファイルは放置すると肥大化し、読み込みトークンが増えてコスト増につながる。定期的な整理を習慣にしよう。

**推奨メンテナンスサイクル:**

| 頻度 | 作業 |
|------|------|
| 毎日 | デイリーノートの確認・追記 |
| 週1回 | デイリーノートから重要事項をMEMORY.mdに転記 |
| 月1回 | MEMORY.mdの古い情報を削除・整理 |
| 半年〜1年 | 古いデイリーノートをアーカイブ/削除 |

`AGENTS.md` のハートビートセクションに定期的なメモリメンテナンスを指示として書いておくと、エージェント自身がこの作業を自動的に行ってくれる:

```markdown
## Memory Maintenance (During Heartbeats)

Periodically (every few days), use a heartbeat to:

1. Read through recent `memory/YYYY-MM-DD.md` files
2. Identify significant events, lessons, or insights worth keeping long-term
3. Update `MEMORY.md` with distilled learnings
4. Remove outdated info from MEMORY.md that's no longer relevant
```

### 📦 コラム: AIに"人格"を与えるということ — SOUL.mdの哲学

OpenClawのユーザーの中には、`SOUL.md` を丁寧に育てることに喜びを見出す人が少なくない。単なる便利ツールではなく、自分だけのキャラクターを持ったパートナーとしてエージェントを作り上げていく——そこには、ちょっと不思議な感覚が伴う。

「AIに人格を与えること」は哲学的な問いでもある。SOUL.mdに書かれた「価値観」や「口調」は、本当の意味でそのAIの"魂"になるのだろうか？答えはおそらく「完全にはならない」だ。LLMは毎回トークンを予測しているだけであり、`SOUL.md` の内容はあくまでプロンプトの一部にすぎない。

しかしそれでも、よく書かれた `SOUL.md` を持つエージェントは、確かに「らしさ」を感じさせる。名前があり、口調があり、得意なことと苦手なことがある。それは読者がそのエージェントと関係を築く上で、重要な役割を果たす。

コンピュータサイエンスの観点から言えば、これは「プロンプトエンジニアリング」の一形態だ。しかし使う側の体験としては、それ以上のものを感じることがある。キャラクターに名前をつけ、価値観を書き込み、何百回も会話を重ねると、そのエージェントはあなたの仕事と日常の一部になっていく。

SOUL.mdを「どう書くか」は自由だ。シンプルに業務向けの設定だけでも十分機能する。しかし少し遊び心を加えて、自分が好きなキャラクターを作ってみると、毎日の使い勝手が大きく変わるかもしれない。AIを「使う」から「育てる」へ——それがOpenClawの面白さの一つだ。

OpenClawの名前の由来でもある「ロブスター（🦞）」は、脱皮（molt）を繰り返して成長する生き物だ。OpenClawの開発者Peter Steinbergerがこの名前を選んだことには、「毎日少しずつ成長し続けるAI」という意味が込められているとも言われている。`SOUL.md` を定期的に見直し、少しずつ更新していくこと——それこそがデジタルな「脱皮」かもしれない。

## まとめ

この章では、OpenClawのメモリシステムの全体像を学んだ。

- **メモリはMarkdownファイル**。透明で、コントロール可能で、ポータブルだ
- **2層構造**: デイリーノート（日次ログ）と `MEMORY.md`（長期記憶）を使い分ける
- **SOUL.md と USER.md** でエージェントのパーソナリティとユーザー像を定義する
- **AGENTS.md** で読み書きの挙動を細かくカスタマイズできる
- **MEMORY.md はプライベートセッション専用**。グループチャットでは読み込まない設計にすること

記憶の仕組みを理解することで、OpenClawは単なるチャットボットから、本当の意味での「パートナー」に進化する。次の章では、このパートナーの能力をさらに拡張する「スキルシステム」について学んでいこう。
