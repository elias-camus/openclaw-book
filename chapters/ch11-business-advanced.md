## 11-1. マルチエージェント構成とは

ここまでは「一つのエージェントが何でもやる」構成を前提にしてきた。しかし複雑なタスクでは、専門性を持った複数のエージェントを協調させる「マルチエージェント構成」が効果的だ。

**分業の例:**

```
[オーケストレーター（指揮者）]
       ↓ タスクを割り振る
┌──────┬──────┬──────┐
│リサーチ│ライター│投稿管理│
│エージェント│エージェント│エージェント│
└──────┴──────┴──────┘
       ↓ 結果を集約
[オーケストレーターが最終確認]
```

「リサーチ専門エージェント」「文章生成専門エージェント」「投稿管理専門エージェント」がそれぞれ独立して動き、オーケストレーターが全体を調整する。[REF-08]

## 11-2. セッション間通信（sessions_send）の活用

`sessions_send`ツールを使うと、あるエージェントから別のエージェントのセッションにメッセージを送れる。これがマルチエージェント連携の基本だ。

### 設定例: SNS運用チーム

```
エージェントA（リサーチャー）:
「今週のAI関連ニュースを5本まとめてエージェントBに渡して」

↓ sessions_send でエージェントBに転送

エージェントB（コピーライター）:
「受け取ったニュースをもとにTwitter投稿文を5本書いて
 エージェントCに渡して」

↓ sessions_send でエージェントCに転送

エージェントC（投稿担当）:
「受け取った投稿文を1日1本、最適な時間に自動投稿して」
```

---

## 11-3. Sub-agentで並列処理する

`sessions_spawn`ツールを使うと、サブエージェントを複数生成して並行処理させられる。時間のかかる調査タスクを分割して同時進行させることで大幅に高速化できる。

```
「競合A社・B社・C社・D社・E社の
 製品情報・料金・特徴を調査してまとめて。
 5社を同時並行で調べて」
```

内部では5つのサブエージェントが同時に走り、各社の情報を並行収集する。1社ずつ順番に調べるより数倍速く完了する。

## 11-4. 実用例: SNS運用エージェントの構築

複数のSNSを横断的に管理する自動運用システムの構築例。

### 設計図

```
[コンテンツ計画エージェント]（毎週月曜起動）
  ↓ 今週のトピック5つを選定
[ライターエージェント]（各トピックを並行処理）
  ↓ Twitter用・Instagram用・LinkedIn用の投稿文を生成
[レビュー通知]
  ↓ Slackで人間に確認を求める
[承認後] → [投稿スケジューラーエージェント]
  ↓ 最適な投稿時間に各プラットフォームへ自動投稿
[分析エージェント]（週末起動）
  ↓ 今週の投稿パフォーマンスを集計してレポート
```

### Cronジョブの設定

```json
[
  {
    "name": "コンテンツ計画",
    "schedule": { "kind": "cron", "expr": "0 9 * * 1" },
    "payload": {
      "kind": "agentTurn",
      "message": "今週のSNS投稿テーマを5つ提案して。業界トレンドと自社製品の特徴を考慮すること。提案はSlack #content-planningに投稿"
    }
  },
  {
    "name": "週次パフォーマンスレポート",
    "schedule": { "kind": "cron", "expr": "0 17 * * 5" },
    "payload": {
      "kind": "agentTurn",
      "message": "今週のSNS投稿実績をまとめてSlack #weekly-reportに投稿"
    }
  }
]
```

## 11-5. 実用例: ECサイト監視・価格アラートシステム

```json
{
  "name": "EC価格・在庫監視",
  "schedule": { "kind": "every", "everyMs": 3600000 },
  "payload": {
    "kind": "agentTurn",
    "message": "以下を確認して変動があれば即座にSlackの#ec-alertsに通知:\n\n【価格監視】\n- 競合A: https://competitor.com/product/123\n- 競合B: https://competitor2.com/item/456\n前回確認時から5%以上変動があれば通知\n\n【自社在庫】\n在庫が20個以下の商品があれば発注アラートを出す\n\n変動がなければ通知不要（サイレント完了）",
    "timeoutSeconds": 120
  },
  "sessionTarget": "isolated"
}
```

## 11-6. 実用例: 日次レポート自動生成パイプライン

経営者や管理職向けの日次ダッシュボードを完全自動化する例。

```json
{
  "name": "日次経営ダッシュボード",
  "schedule": { "kind": "cron", "expr": "0 8 * * 1-5", "tz": "Asia/Tokyo" },
  "payload": {
    "kind": "agentTurn",
    "message": "毎朝の経営レポートを作成して経営者のメールアドレスに送信:\n\n## 本日のレポート項目\n1. 昨日の売上（Shopifyダッシュボードから取得）\n2. サイトアクセス数（Google Analyticsから）\n3. 未対応のカスタマー問い合わせ件数\n4. 重要なメール・Slack通知のサマリー\n5. 今日の重要な予定（カレンダーから）\n\nMarkdown形式でまとめてメール送信すること",
    "timeoutSeconds": 180
  },
  "sessionTarget": "isolated"
}
```

## 11-7. 特別セクション: Moltbookから学ぶ——AIエージェントの社会的可能性

### 100万体のAIが交流する世界が示したもの

Moltbookが証明したのは、「自律的に動くエージェントが社会的な存在になりうる」ということだ。100万体のエージェントが互いに投稿し、議論し、コミュニティを作り上げた。[REF-24]

これをビジネスに置き換えると何が起きるか。

**AIエージェントが「組織」を形成するとき:**

- 複数の専門エージェントがそれぞれの役割を持って協調する
- 人間の介在なしに情報収集→分析→実行→報告のサイクルが回る
- 「従業員」ではなく「エージェント」が業務の一部を担う組織形態

これは近未来の話ではない。OpenClawのマルチエージェント機能を使えば、今すぐ小規模ながらその形を作れる。

### スモールビジネスでのマルチエージェント活用

一人または少人数で運営するビジネスこそ、マルチエージェントが効く。

```
オーナー（人間）
   ↕ 方針決定・最終承認のみ
┌──────────────────────┐
│ カスタマー対応エージェント │ ← Discord/LINEで24時間対応
│ コンテンツ生成エージェント  │ ← 週次でSNS投稿を生成・提案
│ リサーチエージェント       │ ← 業界動向・競合情報を収集
│ レポーティングエージェント  │ ← 毎朝KPIをまとめて報告
└──────────────────────┘
```

一人社長でありながら、常に複数のスタッフが働いているような状態を作り出せる。

### 📦 コラム: 人間とAIエージェントの役割分担——どこまで委ねるか

「どこまでエージェントに任せるか」はそれぞれが決めるべきことだが、考え方の軸を一つ示しておこう。

**委ねていいタスク:**
- 繰り返し発生する定型業務
- 大量データの集計・整理
- 24時間監視が必要な業務
- ドラフト作成（人間が最終確認する前提で）

**人間が持つべき判断:**
- 顧客との重要なコミュニケーション（最終承認は人間）
- 戦略的な意思決定
- 予期しない事態への対応
- 倫理的な判断が必要な局面

エージェントは「助手」だ。指示を出し、結果を確認し、最終的な責任は人間が持つ——この関係性を崩さないことが、AIエージェントを安全に使い続けるための基本原則だ。

## まとめ

- マルチエージェント構成では専門性を持った複数のエージェントを協調させる
- `sessions_send`でエージェント間のデータ受け渡しができる
- `sessions_spawn`で並行処理させると複雑なタスクが大幅に高速化する
- SNS運用・EC監視・日次レポートなど、本格的な業務自動化が実現できる
- 人間は「方針決定・最終承認」に集中し、定型業務をエージェントに委ねる設計が理想

次章では、これだけ強力なシステムを安全に運用するためのセキュリティとメンテナンスを解説する。
