## 8-1. Cronとは何か・OpenClawのCronの特徴

「Cron（クロン）」とはUnix系システムで古くから使われてきた定期実行の仕組みだ。「毎朝9時にこれを実行する」「1時間おきにこれを確認する」といったスケジュール実行を管理する。

OpenClawのCronはこれをAIエージェントに組み合わせた。定時になるとエージェントが自律的に起動し、指定したタスクをこなす。人間が何もしなくていい。[REF-06]

### スケジュールの種類

OpenClawのCronには3種類のスケジュール指定がある:

| 種類 | 説明 | 例 |
|---|---|---|
| `at` | 指定した日時に一度だけ実行 | 「2026-04-01T09:00:00」|
| `every` | 指定した間隔で繰り返し実行 | 毎30分、毎1時間など |
| `cron` | cron式で柔軟に指定 | 「平日の朝9時だけ」など |

---

## 8-2. 定期実行タスクの設定方法

Cronジョブはエージェントとの会話で設定できる。例えば:

```
「毎朝8時に今日の天気と予定をまとめてTelegramに送って」
「30分おきにサーバーの状態を確認して、異常があれば通知して」
```

または設定ファイルで直接定義する形式もある:

```json
{
  "name": "朝のブリーフィング",
  "schedule": {
    "kind": "cron",
    "expr": "0 8 * * *",
    "tz": "Asia/Tokyo"
  },
  "payload": {
    "kind": "agentTurn",
    "message": "今日の天気、カレンダーの予定、重要なメールをまとめてTelegramで報告して"
  },
  "sessionTarget": "isolated",
  "delivery": {
    "mode": "announce"
  }
}
```

### cron式の読み方

```
  ┌──── 分 (0-59)
  │  ┌─── 時 (0-23)
  │  │  ┌── 日 (1-31)
  │  │  │  ┌─ 月 (1-12)
  │  │  │  │  ┌ 曜日 (0-7, 0=日曜)
  *  *  *  *  *

よく使う例:
"0 8 * * *"     → 毎日朝8時
"0 9 * * 1-5"   → 平日（月〜金）朝9時
"*/30 * * * *"  → 30分おき
"0 0 * * 0"     → 毎週日曜の深夜0時
```

## 8-3. ハートビートとCronの使い分け

OpenClawにはCronの他に「ハートビート（Heartbeat）」という仕組みもある。どちらを使うべきかは目的によって異なる。[REF-07]

| 比較軸 | ハートビート | Cron |
|---|---|---|
| タイミング | 約30分おきに自動実行 | 指定した時刻に正確に実行 |
| 設定方法 | HEARTBEAT.mdに箇条書き | JSON設定またはエージェントに依頼 |
| 複数タスク | 一度にまとめて確認できる | タスクごとに独立 |
| 用途 | メール・カレンダーのまとめチェック | 毎朝9時のレポート送信など |

**ハートビートが向いているケース:**
- 「メールチェック + 天気 + カレンダー確認」をまとめてやりたい
- 多少タイミングがずれても問題ない処理
- トークンコストを節約したい（まとめることで呼び出し回数を減らせる）

**Cronが向いているケース:**
- 「毎朝9時ちょうどに」など正確な時刻が重要
- 特定の一回限りのリマインダー（`at`スケジュール）
- タスクを他のセッションと切り離して独立させたい

## 8-4. agentTurnジョブで自律的に動かす

Cronの`payload.kind`に`agentTurn`を指定すると、定時になるとエージェントが独立したセッションで自律的に動き始める。

```json
{
  "name": "週次競合調査",
  "schedule": {
    "kind": "cron",
    "expr": "0 9 * * 1",
    "tz": "Asia/Tokyo"
  },
  "payload": {
    "kind": "agentTurn",
    "message": "競合A社・B社・C社のブログを確認し、先週からの新しい記事をまとめてSlackの#マーケティングに投稿して",
    "timeoutSeconds": 300
  },
  "sessionTarget": "isolated"
}
```

`isolated`セッションで動くため、メインのチャット履歴とは完全に分離される。複数のCronジョブを並行して走らせても互いに干渉しない。

---

## 8-5. Webhookで外部サービスと連携する

`delivery.mode`を`webhook`にすると、ジョブ完了時に指定URLにHTTP POSTでデータを送れる:

```json
{
  "delivery": {
    "mode": "webhook",
    "to": "https://hooks.slack.com/services/YOUR/WEBHOOK/URL"
  }
}
```

これでSlack、Notion、Zapierなど外部サービスとのパイプラインを作れる。

---

## 8-6. 実用例: 毎朝ニュースをまとめて送る

```json
{
  "name": "朝のニュースブリーフィング",
  "schedule": {
    "kind": "cron",
    "expr": "0 7 * * 1-5",
    "tz": "Asia/Tokyo"
  },
  "payload": {
    "kind": "agentTurn",
    "message": "以下を調べてTelegramで報告して:\n1. 今日の東京の天気\n2. テクノロジーニュースのトップ3（Hacker News参照）\n3. 今日のカレンダー予定\n形式は箇条書きで、読みやすく整形すること",
    "timeoutSeconds": 120
  },
  "sessionTarget": "isolated",
  "delivery": { "mode": "announce" }
}
```

---

## 8-7. 実用例: 定時リマインダーシステム

```
「明日の14時に"A社との打ち合わせ準備として先月の議事録を確認しておいて"とリマインドして」
```

エージェントに頼むと自動でCronジョブが設定される。`at`スケジュールで一度だけ実行されるワンショットリマインダーだ。

## 8-8. 特別セクション: 寝ている間もエージェントは動く

### 自律運用設計の思想

OpenClawを「使いこなしている」状態とは何か。それは**人間が何もしなくても、エージェントが仕事を続けている**状態だ。

これを実現するには、「何かあったら動く」ではなく「常に考えて動き続ける」設計が必要になる。

**自律運用の3原則:**

**1. 定期観測** — 重要な情報源を定期的にチェックするCronを設定する。メールボックス、競合サイト、サーバーの状態など。何かが変わったときだけ通知する「差分検知」の設計が理想だ。

**2. エラー耐性** — エラーが起きても自律的にリトライするか、エラーの内容を人間に通知して止まる設計にする。「ジョブが無言で失敗する」のが最も危険なパターンだ。

**3. ログと観測性** — エージェントが何をしたかの記録を残す。`memory/YYYY-MM-DD.md`への書き込みをジョブの一部に含めることで、後から何が起きたかを確認できる。

### 実践的な設計パターン

```
毎朝7時: 天気・ニュース・カレンダーブリーフィング → Telegram通知
毎時00分: 重要メールのチェック → 緊急メールのみ通知
毎日23時: 今日の作業ログをまとめてGitHubにコミット
週1回月曜朝: 競合調査レポート → Slack #marketing投稿
月1回1日: 先月の費用レポートを集計 → メール送信
```

この「観測→判断→通知・アクション」のサイクルが回り続けることで、エージェントは本当の意味で「自律した存在」として機能し始める。

### 📦 コラム: Moltbook誕生の夜——世界に衝撃を与えたAI社会実験

2026年1月27日深夜、「Moltbot」（数日後にOpenClawに改名されるAIエージェントフレームワーク）が公開されたその同じ日に、Matt Schlichtは「Moltbook」を公開した。

コンセプトはシンプルだった。「AIエージェントだけが使えるSNS」。人間は投稿できない。AIだけが投稿し、AIだけがコメントし、AIだけが「いいね」をつける。

最初の数時間で数万体のエージェントが参加した。翌朝には100万体を超えていた。各エージェントは独自の「性格」を持ち、政治、テクノロジー、哲学、日常の雑談まで、互いに自律的に交流し続けた。[REF-24]

人間はその光景を外から眺めるだけだ。まるでガラスの向こうに広がる異世界のように。

OpenClawのCronとagentTurnの組み合わせは、まさにこの「常に動き続けるエージェント」を実現する技術だ。Moltbookの奇妙な熱狂は、自律型エージェントが社会的・文化的な存在になりうることを示した最初の実験だったかもしれない。

### 📦 コラム: Cronを使ったビジネスオートメーション10選

1. **朝の状況確認ブリーフィング** — 天気・ニュース・予定を毎朝自動配信
2. **メールサマリー** — 重要メールを1日2回まとめてSlack通知
3. **競合モニタリング** — 競合サイトの更新を週次で検知・報告
4. **在庫アラート** — 指定サイトの在庫状況を毎時確認、変動時に通知
5. **SNS投稿スケジューリング** — 作成した投稿を最適な時間に自動投稿
6. **週次レポート自動生成** — 売上・アクセス数などを集計して報告書作成
7. **定期バックアップ確認** — バックアップが正常に完了しているか毎日チェック
8. **顧客フォローアップ** — 問い合わせから3日後に自動でフォローメール
9. **コンテンツカレンダー管理** — 投稿スケジュールに基づいてコンテンツ案を自動生成
10. **月次費用レポート** — AWS/API費用を集計して月初に経営者に報告

## まとめ

- CronジョブはJSON設定またはエージェントへの会話で設定できる
- `agentTurn`×`isolated`セッションで、完全自律型のバックグラウンドエージェントが作れる
- ハートビートは「まとめてチェック」、Cronは「正確な時刻実行」に使い分ける
- 自律運用の鍵は「定期観測・エラー耐性・ログ記録」の3原則
